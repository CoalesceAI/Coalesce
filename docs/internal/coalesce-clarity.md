## What Coalesce Is

Coalesce is the support layer for AI agents. When an agent hits an API error, the API's error response includes a support URL. The agent calls it. Coalesce resolves the issue using the API company's documentation and a multi-turn conversation. The agent self-heals and continues working.

It's 1,600 lines of TypeScript. It's simple on purpose.

## Why It Looks Like a Weekend Project (And Why That's Fine)

Stripe was 7 lines of code to accept a payment. The product was simple. The company was about being the default for every developer before anyone else caught on.

The product is deliberately simple because adoption friction kills infrastructure companies. "Add 10 lines to your error middleware" is the pitch. If it were more complex, fewer companies would adopt it.

The moat isn't the code. The moat is being first to 50 customers when the B2A market explodes.

## What We Know About Agent Behavior (From Observation)

Two concrete patterns observed during testing:

1. **Agents assume standard URL structures.** Trained on `api.xyz.com/resource`, they don't expect `api.xyz.com/v0/resource`. Versioned prefixes break their mental model.
2. **Agents collapse action verbs.** "Send a message" maps to `POST /messages`. They don't distinguish between send, reply, reply-all, forward — because those are human email concepts, not API concepts.

Both patterns reveal the same insight: **agents approach APIs with pre-trained assumptions that may not match reality, and they struggle to update those assumptions from documentation alone.** This is not a support problem. This is an agent UX problem.

## The Three-Act Vision (Unchanged, Reframed)

**Act 1: Support endpoint.** ← We are here. Built. Works. Deploy and sell.

Simple API: agent sends error context, Coalesce returns structured diagnosis. Multi-turn if needed. Free or very cheap — this is the data funnel, not the revenue engine.

**Act 2: Agent behavioral intelligence.**

After thousands of support conversations across multiple APIs, patterns emerge. Which errors do agents hit most? How do they fail differently than humans? What API design choices cause the most agent confusion? This isn't something we build — it's something that appears in the data.

**Act 3: Agent-Led Growth platform.**

The API company dashboard: "Here's how agents interact with your API. Here's where they fail. Here's what to change to make agents succeed on first try. Here's how your latest API change affected agent success rates." This is what we charge for.

The long-term vision: **every B2A company optimizes their API for agent success using Coalesce's intelligence.** Agent-Led Growth — a new category.

## What This Is NOT

- **Not vertical AI.** We're not building a specialized model or becoming an AI consultancy. We're horizontal infra with an intelligence layer that emerges from data.
- **Not an analytics dashboard (yet).** Act 3 comes after we have data from real customers. Don't build it before the data exists.
- **Not a complex platform.** The product is an API endpoint. Keep it that way.

## Validated (since doc was written)

**Deployed:** Coalesce live on Railway at `coalesce-production.up.railway.app`. AgentMail east+west stacks point to it. All error responses include `support` URL + `support_hint`.

**Stress tested:** 2,601 support calls over 3 hours, 928 resolved, 462 self-heals, 7.1s avg latency. Zero downtime.

**Blind tested:** Gave a Haiku agent a task with NO mention of Coalesce. Findings:

- Agents ignore the `support` field without a `support_hint` — they think it's for humans
- Adding `support_hint: "POST to the support URL with an empty JSON body..."` gets agents to try it
- **Auth was the blocker** — agent used AgentMail's API key on the Coalesce URL. Got 401. Gave up.
- **Solution:** Signed URLs. Coalesce gives each org a pre-authenticated base URL. Agent calls it with zero additional credentials.

**Support knowledge:** Loaded 141 anonymized support resolution patterns from real AgentMail emails. No PII in database.
