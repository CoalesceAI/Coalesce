## Problem Statement

AI agents are becoming the primary consumers of APIs. When they hit errors, they have no support channel — they retry blindly, give up and escalate to a human, or try to read docs that may not be in their context. The result: developers wake up to 70 stuck agents at 4am over a docs issue that should have been resolved in seconds.

There is no standardized way for an API to help its agent callers self-diagnose errors at the moment of failure. The status quo is "read the docs" — which assumes a human, a browser, and time. Agents have none of these.

## Demand Evidence

- **AgentMail (own company):** Built for AI agents — 100% of API callers are agents. Every error is a machine hitting a wall with no self-service path. Coalesce was built because this pain was firsthand.
- **Browser Use:** Presented the idea to their team. They wanted to try it — asked about integration, not just "interesting." Browser Use's entire product is agents navigating the web; API errors breaking their agents is a direct pain point.
- **Market signal:** YC has explicitly stated interest in B2A (Business-to-Agent) companies. The B2A category is recognized and named. Gartner predicts 90% of B2B purchases intermediated by agents by 2028.
- **Discourse signal:** Articles appearing on "making APIs agent-ready," "self-healing agent infrastructure," and "error handling for AI agents in production" — the problem is being named, but nobody has productized the solution.

## Status Quo

When an agent hits an API error today, what happens depends entirely on the agent's implementation:
- **Blind retry:** Wastes tokens and time, often fails again
- **Escalate to human:** Defeats the purpose of having an agent; developer debugs manually
- **Read docs (if available):** Unreliable — requires docs in context window, slow, agent may hallucinate fixes
- **Give up:** Agent stops, work is lost, human discovers the failure hours later

No API currently provides a machine-readable path to resolution at error-time. The closest analog is Stripe's error documentation (each error code links to docs), but that's static content for humans, not a conversational resolution endpoint for agents.

## Target User & Narrowest Wedge

**Target user:** Founders and engineering leads at B2A companies — companies whose APIs are primarily consumed by AI agents. Examples: AgentMail (email for agents), Browser Use (web browsing for agents), agent framework companies (LangChain, CrewAI, Composio), MCP server providers.

**What gets them promoted:** High API reliability, low support burden, developer/agent satisfaction metrics.

**What gets them fired:** Agents failing silently at scale, support tickets piling up, developers abandoning the API because agents can't use it reliably.

**Narrowest wedge:** A company signs up, uploads their docs (or points to a URL), and gets a support endpoint. They add 3 lines of code to their error middleware: when an error occurs, include a `support` URL in the response pointing to Coalesce. That's it. Coalesce handles the conversation, the resolution, and the multi-turn follow-up. The API company writes zero AI code.

## Constraints

- Must work with any API — not tied to AgentMail
- Must be trivially easy to integrate (<30 minutes from sign-up to first resolution)
- Resolution must be grounded in retrieved documentation with source references. When no relevant docs are found, return "unresolvable" rather than guessing
- Must handle multi-turn conversation via session IDs (agent may need to provide clarifying info)
- Must return structured JSON conforming to a defined schema so agents can programmatically act on the fix
- v1: HTTP only. WebSocket deferred to v2 (already built in current prototype, can be promoted later)

## Premises

1. **AI agents are becoming primary API consumers.** When they fail, there's no support channel — they just stop. This is a real, accelerating trend.
2. **The fix is trivially easy for the API company.** Minimal code to route errors to Coalesce. Toll-free model — low friction adoption.
3. **Coalesce owns the resolution conversation, not the API company.** The company provides the knowledge; Coalesce provides the AI, the infra, and the conversation management.
4. **Data is the compounding asset.** Every resolution makes the system smarter (specialized model) and gives API companies product insights they can't get anywhere else (analytics).
5. **The "support URL in error responses" pattern is a new primitive.** Nobody does this. It's the wedge that makes everything else possible.

**Plain is the closest threat.** They're API-first, they see agents coming (MCP + BYOA), they have great customers (Vercel, Cursor). But their architecture is "human reads agent's work" — not "agent self-heals." If Plain pivots to offer error-time agent support, they'd be the main competitor. Speed matters.

## Approaches Considered

### Approach A: Agent Support Endpoint (Wedge-first) — CHOSEN
Ship Act 1 as a standalone product. Any API company signs up, uploads docs, gets a support endpoint. Focus on 5-10 B2A companies. No dashboard, no analytics — just flawless resolution.

- Effort: M (human: ~2 months / CC: ~2 weeks)
- Risk: Low
- Pros: Fastest to market; core already built; wedge is sharp ("add 3 lines, agents self-heal")
- Cons: No data moat yet; no dashboard for API company decision-makers

### Approach B: Full Three-Act Platform
Build all three acts in parallel — knowledge lake, resolution, data collection, analytics dashboard.

- Effort: L (human: ~6 months / CC: ~6 weeks)
- Risk: High
- Pros: Complete fundraising story; analytics is what gets champions promoted
- Cons: Risk of building before validating; spreading focus across three acts

### Approach C: Protocol Play
Define an open spec for "agent support URLs in error responses." Coalesce is the default implementation.

- Effort: L (human: ~4 months / CC: ~4 weeks)
- Risk: Medium-High
- Pros: Network effects; if you define the standard, you're the default
- Cons: Standards adoption is slow; competitors could implement without using Coalesce

## Recommended Approach

**Approach A: Wedge-first.** Ship Act 1 as a standalone product and prove demand with real B2A companies. The three-act vision is the fundraising narrative — Act 1 alone is the thing that either works or doesn't.

**What to build:**
1. **Multi-tenant knowledge store** — per-tenant document storage. Accepts URL crawl, PDF upload, markdown, and OpenAPI specs. Documents are chunked and stored for retrieval at diagnosis time (vector store or full-context injection depending on corpus size per tenant)
2. **Support endpoint per tenant** — `POST /support/{tenant}` (see API contract below)
3. **Integration guide** — code snippets for error middleware in Node, Python, Go ("add support URL to your error responses")
4. **Manual onboarding for first 5-10 customers** — founder onboards each one personally. Self-serve deferred until demand is validated

**API Contract:**

Initial request (agent hits the support URL from the error response):
```json
POST /support/{tenant}?endpoint=/v0/inboxes/{id}/messages&error_code=404&context=NotFoundError
Body (optional): { "tried": ["used a random inbox ID"] }
```

Response:
```json
{
  "status": "needs_info",
  "session_id": "abc-123",
  "turn_number": 1,
  "question": "What inbox ID did you use?",
  "need_to_clarify": ["inbox_id value", "whether inbox was created first"]
}
```

Follow-up:
```json
POST /support/{tenant}
Body: { "session_id": "abc-123", "answer": { "clarifications": { "What inbox ID did you use?": "nonexistent-inbox-id" } } }
```

Resolution:
```json
{
  "status": "resolved",
  "session_id": "abc-123",
  "turn_number": 2,
  "diagnosis": "The inbox 'nonexistent-inbox-id' does not exist.",
  "fix": {
    "description": "Create the inbox first, then send the message.",
    "fix_steps": [
      { "action": "POST /v0/inboxes to create a new inbox" },
      { "action": "Use the returned inbox_id in your message request" }
    ]
  },
  "references": ["Inboxes API > Create Inbox"]
}
```

When no docs cover the error: `{ "status": "unknown", "explanation": "..." }`. Structured output is enforced via Zod schema validation on the LLM response (already implemented in prototype).

**Authentication:** API key per tenant, passed via `Authorization: Bearer` header. Rate limiting per key (TBD limits).

**When Coalesce is down:** The support URL is a best-effort enhancement. Integration guide recommends: if the support endpoint returns non-200 or times out, the agent should fall back to its normal error handling. Coalesce failure should never block the agent's primary workflow.

**What NOT to build yet:**
- Self-serve onboarding UI — defer until 10+ customers
- Analytics dashboard (Act 3) — defer until 5+ customers
- Specialized model (Act 2) — defer until enough resolution data
- Protocol spec (Approach C) — defer until the pattern is proven
- WebSocket transport — already built in prototype, promote when a customer needs it

## Open Questions

1. **Pricing model:** Usage-based, per-resolution. **"30-second resolution or you don't pay."** Pizza Hut guarantee — if Coalesce doesn't resolve the agent's error within 30 seconds, it's free. Only charge for successful resolutions. This is the GTM hook: zero risk for the API company, Coalesce only makes money when it delivers value. Exact per-resolution price TBD ($0.50-$1.00 range).
2. **Knowledge freshness?** Crawl on schedule? Webhook-triggered re-index? Both?
3. **Multi-region?** Agents are global. Does Coalesce need to be?
4. **What if the docs don't cover the error?** Return "unknown" + escalate to human support (if configured)? This is the Plain/Pylon handoff point.
5. **How do you get the first 5 customers?** Browser Use is warm. Who else? Agent framework companies (LangChain, CrewAI)? MCP server providers?

## Success Criteria

1. 5 B2A companies using Coalesce in production within 3 months
2. Measurable reduction in human-escalated agent errors for at least 1 customer
3. Resolution rate >60% — defined as: the agent's subsequent API call succeeds after applying the fix. Measured by tracking whether the same agent+endpoint+error_code combo recurs within the session
4. Integration takes <30 minutes from sign-up to first resolution
5. At least 1 customer who would be "genuinely upset if it disappeared tomorrow"
