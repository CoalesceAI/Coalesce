## Problem Statement

Coalesce has a working multi-tenant prototype. The next step is evolving it from demo to product — but without over-engineering. The previous design doc proposed org/pod hierarchies, S3 knowledge lakes, URL crawlers, usage tracking, and caching layers. This revision strips it back to what's actually needed: the minimal architecture that supports real customers.

## What Changed (and Why)

The previous design was premature. Key realizations:

- **No pods.** One org = one set of docs. AgentMail doesn't need separate pods for HTTP vs WebSocket. No customer has asked for this. Add it when someone does.
- **No caching.** Load docs from DB on every request. The data is small (<200KB), the query is fast, and caching adds complexity for zero benefit at current scale.
- **No usage tracking.** Remove the usage table. Add it back when billing matters.
- **No S3.** Docs live in Postgres. No processing pipeline. Fetch and serve.
- **Rename tenants to organizations.** Matches Clerk (our auth provider) and industry convention.
- **Add external_customer_id to sessions.** So API companies can track which of THEIR customers' agents are hitting errors. Coalesce owns the analytics long-term.

## Architecture

```
Agent hits API error → error includes support URL
                    → agent POSTs to /support/{org}
                    → Coalesce auth middleware validates API key
                    → load org's docs from Postgres
                    → send to Claude with error context
                    → multi-turn conversation if needed
                    → return structured resolution
```

No cache, no S3, no pods, no usage table. Request → DB → Claude → response.

## Schema

```sql
-- Organizations: customer companies (renamed from tenants)
organizations (
  id UUID PK DEFAULT gen_random_uuid(),
  slug TEXT UNIQUE NOT NULL,       -- "agentmail", "browseruse"
  name TEXT NOT NULL,              -- "AgentMail"
  settings JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
)

-- API keys: authenticate requests to an org
api_keys (
  id UUID PK DEFAULT gen_random_uuid(),
  org_id UUID FK → organizations ON DELETE CASCADE,
  key_hash TEXT UNIQUE NOT NULL,
  prefix TEXT NOT NULL,
  label TEXT,
  created_at TIMESTAMPTZ DEFAULT now(),
  revoked_at TIMESTAMPTZ
)

-- Doc sources: where docs come from
doc_sources (
  id UUID PK DEFAULT gen_random_uuid(),
  org_id UUID FK → organizations ON DELETE CASCADE,
  source_type TEXT CHECK (source_type IN ('url_crawl','file_upload','github','notion','openapi','manual')),
  config JSONB DEFAULT '{}',
  status TEXT CHECK (status IN ('pending','crawling','processing','ready','error')) DEFAULT 'pending',
  last_sync_at TIMESTAMPTZ,
  error_message TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
)

-- Doc content: processed text for Claude context injection
doc_content (
  id UUID PK DEFAULT gen_random_uuid(),
  org_id UUID FK → organizations ON DELETE CASCADE,
  source_id UUID FK → doc_sources ON DELETE CASCADE,
  title TEXT,
  content TEXT NOT NULL,
  metadata JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
)

-- Sessions: support conversations
sessions (
  id UUID PK DEFAULT gen_random_uuid(),
  org_id UUID FK → organizations ON DELETE CASCADE,
  external_customer_id TEXT,      -- API company's customer ID (optional)
  turns JSONB DEFAULT '[]',
  original_request JSONB NOT NULL,
  status TEXT CHECK (status IN ('active','resolved','expired','abandoned')) DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT now(),
  last_accessed_at TIMESTAMPTZ DEFAULT now(),
  resolved_at TIMESTAMPTZ
)
```

5 tables. No usage, no pods. Clean.

## Endpoint

```
POST /support/{org_slug}
Authorization: Bearer clsc_live_...
Content-Type: application/json

Query params (initial): ?endpoint=...&error_code=...&context=...&customer_id=...
Body (follow-up): { "session_id": "...", "answer": { "clarifications": {...} } }
```

`customer_id` is optional — passed by the API company's error middleware from their auth context. Stored as `external_customer_id` on the session.

## Auth Flow

1. Extract Bearer token → 401 if missing
2. Hash token, lookup in api_keys → 401 if not found or revoked
3. Resolve org from key → verify URL org_slug matches → 403 if mismatch
4. Load org's docs from doc_content (fresh query, no cache)
5. Run diagnosis

## Migration from Current Schema

The current DB schema (from migrations 001-004) differs from the target schema above. The migration reconciles them in one shot.

```sql
-- 005_organizations_and_cleanup.sql

-- 1. Rename tenants → organizations
ALTER TABLE tenants RENAME TO organizations;
ALTER INDEX idx_tenants_slug RENAME TO idx_organizations_slug;

-- 2. Rename tenant_id → org_id on all tables
ALTER TABLE api_keys RENAME COLUMN tenant_id TO org_id;
ALTER TABLE doc_sources RENAME COLUMN tenant_id TO org_id;
ALTER TABLE doc_content RENAME COLUMN tenant_id TO org_id;
ALTER TABLE sessions RENAME COLUMN tenant_id TO org_id;

-- 3. Add settings to organizations (if not exists)
ALTER TABLE organizations ADD COLUMN IF NOT EXISTS settings JSONB DEFAULT '{}';

-- 4. Fix api_keys: add prefix, add revoked_at, migrate active→revoked_at
ALTER TABLE api_keys ADD COLUMN IF NOT EXISTS prefix TEXT DEFAULT '';
ALTER TABLE api_keys ADD COLUMN IF NOT EXISTS revoked_at TIMESTAMPTZ;
UPDATE api_keys SET revoked_at = now() WHERE active = false;
ALTER TABLE api_keys DROP COLUMN IF EXISTS active;

-- 5. Sessions: add external_customer_id, last_accessed_at, resolved_at
ALTER TABLE sessions ADD COLUMN IF NOT EXISTS external_customer_id TEXT;
ALTER TABLE sessions ADD COLUMN IF NOT EXISTS last_accessed_at TIMESTAMPTZ DEFAULT now();
ALTER TABLE sessions ADD COLUMN IF NOT EXISTS resolved_at TIMESTAMPTZ;

-- 6. Drop usage table
DROP TABLE IF EXISTS usage;
```

## What We're NOT Building

- Pods (add when a customer needs multiple products)
- S3 storage (add when docs exceed Postgres comfort zone)
- Caching (add when request latency from DB becomes a problem)
- Usage tracking (add when billing matters)
- URL crawler (add when manual doc loading isn't enough)
- Notion/GitHub integrations (add when a customer asks)

Each of these is a future migration measured in hours, not weeks. None block customer onboarding.

## Clerk Integration (Future)

Clerk manages organizations, members, and auth. Currently we use custom `clsc_live_*` keys. Future path: Clerk-issued API keys, Clerk org IDs as our org identifiers, Clerk webhooks for org lifecycle events. This is a swap, not a rebuild — the org model is already aligned.

## Premises

1. **The current schema is 90% right.** Rename, add one field, drop one table. That's the whole migration.
2. **No pods, no cache, no S3, no usage.** Each is cheap to add later. None are needed now.
3. **Organizations (not tenants) matches Clerk and customer mental models.**
4. **external_customer_id is the one new field that matters.** It enables Act 3 analytics without a schema change later.
5. **The next step after this migration is onboarding a second customer.** Not more architecture.

## Success Criteria

1. Migration runs without errors
2. AgentMail works as before (org slug: "agentmail")
3. `/support/{org}` endpoint works with org-based auth
4. Sessions store external_customer_id when provided
5. TypeScript compiles, all tests pass
